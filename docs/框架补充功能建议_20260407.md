# Playwright UI 框架补充功能建议（实战可用性）

生成日期：2026-04-07  
分析范围：`playwright01` 核心执行链路（`conftest.py`、`run.py`、`module/`、`utils/`、`testcases/`）

## 1. 当前已有能力（做得不错）
- 基于 `pytest + playwright` 的页面对象模式已成型。
- 已有失败截图、视频、trace、重跑策略、Allure 报告。
- 有慢接口统计与历史趋势（`conftest.py` 中的 API 性能记录逻辑）。
- 有飞书通知和失败自动提缺陷（`utils/feishu_robot.py`、`utils/bug_helper.py`）。

## 2. 实际工作中建议优先补充的功能

### P0（优先补）

#### 2.1 配置中心与密钥托管
现状：`globalconfig.py`、`run.py` 存在 URL、Webhook、账号等硬编码。  
建议补充：
- 增加 `config/` 分层配置：`base.yaml` + `env/dev.yaml` + `env/test.yaml`。
- 使用 `.env` 或 CI Secret 注入账号/Token/Webhook。
- 启动时打印“脱敏后的运行配置快照”（便于排查环境错配）。

价值：降低泄漏风险，支持多人/多环境稳定复用。

#### 2.2 统一命令行入口（单一 run 入口）
现状：`run.py` 与 `run2.py` 职责重叠，执行方式分散。  
建议补充：
- 新增统一 CLI：
  - `python -m playwright01.cli run --env test --suite cat2bug --headed false`
  - `python -m playwright01.cli report`
  - `python -m playwright01.cli notify`
- 将“清理目录、执行测试、生成报告、消息通知”拆为独立子命令。

价值：本地与 CI 指令一致，减少口口相传和误操作。

#### 2.3 失败根因聚类与稳定性看板
现状：当前可收集失败记录，但缺少自动归因。  
建议补充：
- 对失败信息按关键词聚类：`UI定位失败`、`环境网络异常`、`断言失败`、`权限问题`。
- 每次运行生成 `stability_summary.json`，输出 Top 失败类型和波动趋势。
- 对同一 `nodeid` 连续失败次数做统计，自动标记“疑似脆弱用例”。

价值：让团队先修最影响交付的稳定性问题。

### P1（建议尽快补）

#### 2.4 用例标签治理与测试集编排
现状：有 `markers`，但业务维度编排能力不足。  
建议补充：
- 统一标签规范：`smoke`、`regression`、`api_related`、`p0`、`serial`、`parallel`。
- 增加“测试集清单文件”（例如 `suites/*.yaml`），由清单控制执行范围，而不是手写路径。

价值：支持“冒烟 10 分钟回归”“发布前全量”等固定流程。

#### 2.5 Test Data Factory + 数据回收机制
现状：`data_module` 有静态数据，但动态测试数据生成与清理不足。  
建议补充：
- 增加 `utils/data_factory.py`，统一生成用户名、订单号、手机号等。
- 为新增数据打“测试前缀+运行ID”，结束后自动回收（API 或 DB 清理）。

价值：减少脏数据与串测，提升并行运行可靠性。

#### 2.6 登录态缓存策略标准化
现状：`PageIns.new_context_return_page*` 已有 `storage_state` 思路，但行为分散且重复。  
建议补充：
- 统一 `auth/session_manager.py`：按环境+角色管理缓存、过期时间、强制刷新。
- 合并 `new_context_return_page` / `_mall` / `_cat` 为可配置策略。

价值：降低重复代码，提升登录相关失败可控性。

### P2（中期增强）

#### 2.7 可观测性增强（链路级）
建议补充：
- 将 UI 操作和网络请求关联到同一 `trace_id`。
- 失败时自动附加“最近 N 条 API 请求摘要”（URL、状态码、耗时、响应片段）。
- 输出机器可读事件日志 `logs/events/*.jsonl`。

价值：定位问题从“看视频猜”升级为“结构化证据定位”。

#### 2.8 CI 模板与质量门禁
建议补充：
- 提供 Jenkins/GitLab/GitHub Actions 模板。
- 门禁规则：
  - Smoke 失败直接阻断；
  - 新增用例必须带标签；
  - 关键模块成功率低于阈值时告警。

价值：从“能跑”升级到“能持续稳定交付”。

#### 2.9 编码与文档标准化
现状：部分文件存在乱码（编码不统一）和示例代码残留。  
建议补充：
- 统一 UTF-8 编码与换行规范。
- 增加贡献文档：目录约定、命名规范、PageObject 编写模板、Fixture 设计规范。

价值：降低新人接入成本，避免维护质量波动。

## 3. 推荐落地顺序（4 周）
- 第 1 周：配置中心、密钥托管、统一 CLI。
- 第 2 周：失败聚类看板、标签治理、测试集编排。
- 第 3 周：数据工厂、登录态管理器、脆弱用例识别。
- 第 4 周：CI 模板、门禁策略、文档和编码治理。

## 4. 最小可交付（建议先做这 3 件）
- 统一 CLI 入口。
- 配置中心 + Secret 注入。
- 失败根因聚类报表。

这三项完成后，框架在团队内的可复制性和稳定性会明显提升，后续再逐步补充数据与观测能力，投入产出比最高。
